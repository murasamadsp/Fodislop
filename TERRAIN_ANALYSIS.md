# Разбор поломки рендеринга террейна

Анализ цепочки багов: от `TentaclePool` до исчезающего террейна и параллакса.
Статус: расследование не завершено, но собрана полная картина.

## 2. Баг №1 — TentaclePool (решён)

**Симптом:** при закрытии сцены Unity ругался `Some objects were not cleaned up when closing the scene. [TentaclePool]`.

**Причина:** `TentaclePool` создавал корневой GameObject `[TentaclePool]` с `Object.DontDestroyOnLoad(go)`. Этот объект переживал выгрузку сцены, а метод `Clear()` нигде не вызывался → утечка.

**Фикс:** убран `DontDestroyOnLoad`. Объект теперь уничтожается вместе со сценой, а при следующем обращении пересоздаётся (Unity-овский `== null` корректно отрабатывает на уничтоженном объекте). Файл: `Assets/Scripts/Game/TentaclePool.cs`.

## 3. Баг №2 — террейн исчезает после полной загрузки

**Симптом:** заходишь в игру → террейн подгружается клетка за клеткой → полностью построился → **мгновенно пропадает**, остаётся голубой фон (clear color камеры). Происходит само по себе, без ввода. Появился **после рефакторинга `adb4708`**, до моих правок.

### Что именно сломал рефакторинг `adb4708` («massive SOLID рефакторинг»)

Коммит порезал `SingleMeshTerrainRenderer.cs` на 554 строки (в основном удаления) и вынес логику в новый файл. Плюс есть некоммиченные правки поверх. Ключевые изменения:

#### 3.1. Флуд-филл вынесен в `BackgroundFloodFill.cs` (новый файл, 469 строк)
Раньше `ComputeBackgroundMap` / FBPW жили прямо в рендерере. Теперь — отдельный класс с интерфейсом `ICachedCellDataProvider`. Логика вынесена корректно (границы массивов проверил — не вылезает), это **не** главный подозреваемый, но добавило точку отказа.

#### 3.2. Удалён инкрементальный скролл — теперь всегда `FullRebuild`
Было:
```csharp
if (canScroll) IncrementalUpdate(...);   // сдвиг вершин in-place, дёшево
else             FullRebuild(...);        // полный Clear + пересборка
```
Стало:
```csharp
FullRebuild(minX, minY, atlases);         // ВСЕГДА Clear + пересборка
```
`FullRebuild` → `BuildMeshData(clearMesh: true)` → **`_mesh.Clear()`** каждый раз. Раньше полный Clear случался только на первом кадре / больших прыжках. Теперь — при любом движении камеры и любой загрузке текстуры. Это и регрессия производительности, и смена поведения.

#### 3.3. `OnTextureLoaded` стал убивать материалы при каждой загрузке
В обработчик добавили:
```csharp
_atlasIndexCache.Clear();
_lastAtlasCount = -1;   // ← форсит CleanupMaterials() на следующем кадре
```
`_lastAtlasCount = -1` гарантирует, что в `UpdateVertexAttributes` условие `atlases.Count != _lastAtlasCount` всегда истинно → вызывается `CleanupMaterials()` (**Destroy всех материалов**) → пересоздание. **Причём `WorldTextureManager.OnTextureLoadedHandler` форвардит ВСЕ ассеты** (скины, хвосты, паки, кристаллы, клановые иконки — не только `cells/*`), значит **каждая загруженная текстура любого типа** триггерит полную пересборку террейна с уничтожением материалов.

#### 3.4. Материалы теперь перезаписываются каждый кадр
В `BuildMeshData` убрали guard `if (_materials[i].GetTexture("_BaseMap") != atlasTex)`. Теперь `SetVector/SetColor/SetTexture/SetFloat` вызываются безусловно на каждой пересборке. Функционально безвредно, но добавляет GPU-работы.

#### 3.5. Шейдер: переставлен порядок `discard`
Было: `if (color.a < 0.05) discard;` → `if (subAtlasRect.z < 0.0001) return color;`
Стало: `if (subAtlasRect.z < 0.0001) return color;` → `if (color.a < 0.05) discard;`
Ячейки без текстуры (fallback) теперь возвращают цвет **до** проверки альфы. Вместе с серой заливкой в `FillQuadData` — больше ячеек рендерится, не меньше. К исчезновению напрямую не ведёт, но меняет картинку.

#### 3.6. Потенциальный `DivideByZeroException`
В `UpdateVertexAttributes`:
```csharp
var atlases = wtm.GetAllAtlases();
if (atlases == null) return;              // ← проверка на null, но НЕ на Count == 0
...
int estimatedPerAtlas = (... / atlases.Count) + 16;  // ← /0 если список пуст
```
Если `atlases` — пустой список (не null), будет `DivideByZeroException`. Маловероятно (`EnsureInitialized` всегда держит ≥1 атлас), но дыра есть.

## 4. Ключевая зацепка — `OnTextureLoaded` может вообще не вызываться

Я добавил `Debug.Log` в `OnTextureLoaded` рендерера и собрал билд. В твоём логе (`[MapManager]`, `[Robot]`, `[PersistentAssetCache]`, `[WorldTextureManager] Successfully loaded texture for Empty cell`) **нет ни одной строки `[Terrain]`**, хотя текстуры ячеек грузятся. Два варианта:

1. **Ты запустил старый билд** (без моих логов) — тогда вывод нельзя делать.
2. **`OnTextureLoaded` реально не доходит** до рендерера.

Вариант 2 правдоподобен из-за синглтон-гонки:

```csharp
// SingleMeshTerrainRenderer.Awake()
if (WorldTextureManager.Instance != null)           // ← .Instance АВТОСОЗДАЁТ синглтон
{
    WorldTextureManager.Instance.OnTextureLoaded += OnTextureLoaded;
}
```
`SingletonMonoBehaviour.Instance` при первом обращении **создаёт новый GameObject**. Если рендерер дергает `.Instance` раньше, чем создан «настоящий» `WorldTextureManager`, рендерер подписывается на **временный** инстанс. Если потом появляется другой инстанс менеджера — события идут в него, а рендерер сидит на мёртвой подписке.

**Что проверить:** залогировать `WorldTextureManager.Instance.GetInstanceID()` в `Awake()` рендерера и в месте, где реально грузятся текстуры. Если ID разные — это и есть корень.

## 5. Почему террейн «исчезает навсегда», а не мерцает

После последней загруженной текстуры:
- больше нет `OnTextureLoaded` → `_needsRefresh` остаётся `false`;
- камера не двигается → `currentGridPos == _lastGridPos`;
- размеры не меняются.

Значит **после последней пересборки меш заморожен**. Если эта последняя пересборка вернула пустой/невидимый меш — он таким и останется. Это объясняет, почему не мерцание, а именно «построился → пропал и не вернулся». Виновник — то состояние, в котором оказалась последняя `FullRebuild`:
- либо `PopulateCellCache` молча вернулся (если `MapStorage.IsReady` в тот кадр был `false`) → весь кэш `CellType.Empty`/`Unloaded` → невидимо;
- либо материалы/атласы в тот кадр были в промежуточном состоянии;
- либо `_mesh.Clear()` прошёл, а загрузка вершин/индексов не досчиталась (исключение — но try/catch в `BuildMeshData` его логирует, а в логе ошибок нет).

## 6. Баг №3 — параллакс (новый симптом)

После частичных правок террейн появился, но пошёл параллакс (террейн движется не синхронно с камерой). Наиболее вероятная связь — **удаление инкрементального скролла** (п. 3.2):

- Трансформ меша выставляется только при смене целочисленной `currentGridPos`:
  ```csharp
  transform.position = new Vector3(currentGridPos.x * _cellSize, currentGridPos.y * _cellSize, 0);
  ```
- Камера движется плавно, трансформ — дискретными прыжками по сетке.
- Раньше `IncrementalUpdate` + `ScrollVertexBuffer` компенсировали подсеточное смещение, сдвигая вершины. Теперь этого нет — каждый кадр полный ребилд по целочисленной сетке, и террейн «трясётся»/отстаёт относительно плавной камеры.

То есть параллакс — это **прямое следствие удаления инкрементального скролла**, а не отдельный баг.

## 7. Что предлагаю делать (по приоритету)

1. **Вернуть инкрементальный скролл** (`IncrementalUpdate` + `ScrollVertexBuffer`) — уберёт и параллакс, и лишние `_mesh.Clear()`. Это самая большая функциональная регрессия рефакторинга.
2. **Убрать `_lastAtlasCount = -1` из `OnTextureLoaded`** — не уничтожать материалы на каждую текстуру. Очистки `_metadataCache`/`_atlasIndexCache` + `_needsRefresh` достаточно, материалы пересоздадутся сами при реальной смене числа атласов.
3. **Отфильтровать `OnTextureLoaded`** в `WorldTextureManager.OnTextureLoadedHandler` — форвардить только `cells/*`, а не скины/хвосты/паки. Иначе каждый ассет пересобирает террейн.
4. **Починить подписку на синглтон** — подписываться не в `Awake()` через авто-создание, а лениво при первом `LateUpdate`, когда `WorldTextureManager.Instance` гарантированно настоящий; проверить `GetInstanceID()`.
5. **Добавить проверку `atlases.Count == 0`** перед делением (п. 3.6).
6. Прогнать с `[Terrain]`-логами ещё раз и посмотреть, что происходит **в кадр исчезновения** (пришли лог, если запускал именно свежий билд — там должны быть строки `[Terrain]`).
