# UI Scaling Audit — что понял

> Разбор перед переписыванием. Кода не трогал. Дата: 2026-05-16.

## Симптом

UI «то мелкое, то огромное» на разных DPI/PPI/платформах (MacBook Retina,
4K, мобилка, iPad). Смена `PanelSettings.ScaleMode` с `ConstantPhysicalSize`
на `ScaleWithScreenSize` (коммит 8e79f87, **откатан** в b2558a6) сделала
**хуже**: layout стал хаотичным, кнопки разъехались.

## Корневая причина — НЕ в PanelSettings

Это не проблема настройки скейл-режима. **У интерфейса вообще нет
адаптивной модели лейаута.** Он построен на абсолютных пиксельных
координатах + хардкод-px + ноль USS:

### 1. Сервер диктует абсолютные пиксели

`Assets/Scripts/PacketUIBuilder.cs:42-63` — `ApplyAttachedProperties`
читает из серверного пакета `Canvas.X / Canvas.Y / Canvas.Width /
Canvas.Height` и ставит **сырыми пикселями**:

```
element.style.position = Position.Absolute;
element.style.left  = <raw px от сервера>;
element.style.top   = <raw px от сервера>;
element.style.width = <raw px>;  element.style.height = <raw px>;
```

Весь динамический UI позиционируется абсолютом в экранных пикселях,
которые задаёт сервер. Ни якорей, ни относительных единиц, ни flex.
`Assets/Scripts/UI/Builders/GridPacketBuilder.cs:136-138` — то же
(`Position.Absolute; left; top`).

### 2. Все размеры/отступы/шрифты — сырые px

- `Assets/Scripts/UI/Builders/StyleApplicator.cs` — `borderWidth /
  margin / padding` применяются как сырые числа (px), без масштаба.
- Хардкод шрифтов: `FPSCounter.cs:46` `fontSize = 14`,
  `MinimapPlaceholder.cs:174` `fontSize = 20`.
- `Assets/Resources/UI/MainMenu.uxml` — кнопка `width:125px;
  height:35px; top:100px`.
- Относительные единицы (`%`, `Length.Percent`) — только у пары корней
  (`MainMenu.cs:58-59,80-81`), больше нигде.

### 3. Нет USS / дизайн-системы

Ни одного `.uss` в проекте. Единственная тема — дефолтная рантайм
(`@import url("unity-theme://default")`). Вся стилизация — инлайн
`element.style.X = ...` из C#. Нет центрального стайлшита, токенов
(шкала шрифта/отступов), брейкпоинтов.

### 4. MainMenu — абсолютные костыли

`Assets/Scripts/MainMenu.cs:44-51` — лоадер и меню принудительно
`Position.Absolute` на весь экран + `BringToFront` (это мой же фикс
z-order ранее — функциональный пластырь, не лейаут).

## Почему ScaleWithScreenSize всё сломал

`ScaleWithScreenSize` умножает координаты на коэффициент относительно
reference-разрешения. Но layout — это **абсолютные пиксельные позиции
от сервера, посчитанные под некий подразумеваемый размер канваса**.
При смене коэффициента абсолютные дети репозиционируются/ресайзятся
неравномерно относительно (тоже отмасштабированных, но иначе
заякоренных) родителей → элементы, которые «случайно смотрелись
нормально» при `ConstantPhysicalSize` (там пиксель ≈ пиксель, серверные
координаты «попадали»), разъезжаются. Никакого лейаута, который можно
масштабировать, нет — есть только абсолютные позиции.

**Вывод:** ни одно значение `m_ScaleMode` не чинит «хаотично +
разъехалось», потому что масштабировать нечего. Инстинкт «надо
переписывать» — верный для настоящей кроссплатформенности.

## Открытый вопрос (решает архитектуру, нужен ответ до переписывания)

**Обязан ли сервер слать абсолютные `Canvas.X/Y/W/H`?**

- Если **да** (протокол фиксирован): клиент должен трактовать их как
  координаты в **design-space** против фиксированного reference, а
  абсолютные дети должны жить под контейнером фиксированного
  design-размера (а не под экранным корнем), и уже этот контейнер
  единообразно масштабировать через `ScaleWithScreenSize`.
- Если сервер **может** слать относительный/flex-лейаут — это
  радикально лучше; абсолют убираем почти везде.

## План переписывания (фазы, каждая — план → ревью → код → Play-проверка)

1. **Решить контракт с сервером** (вопрос выше). Без этого остальное
   гадание.
2. **Design-space контейнер.** Один корень фиксированного размера
   (напр. 1920×1080), абсолютные серверные координаты — внутри него;
   `PanelSettings = ScaleWithScreenSize` масштабирует контейнер целиком
   единообразно.
3. **Builders → flex/relative**, где сервер это допускает
   (Panel/Dock/Grid/ScrollViewer вместо `Position.Absolute`).
4. **USS + токены.** Вынести инлайн-стили в стайлшит; шкала шрифта и
   отступов в относительных единицах вместо хардкод-px
   (`StyleApplicator`, `FPSCounter`, `MinimapPlaceholder`, MainMenu).
5. **Только потом** настройка `ScaleWithScreenSize` (reference + Match)
   — она заработает как задумано, когда под ней будет реальный лейаут.

## Объём и риск

Это не правка настроек — это рефактор слоя UI: `PacketUIBuilder` +
все ~14 builders + новый USS + решение по контракту с сервером.
Поведение-меняющее, обязательна итеративная Play-проверка на наборе
экранов. Делать фазами с ревью, не блайнд-патчем (урок B/скейлинга).
